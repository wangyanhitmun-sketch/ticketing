## 1. 冻结目标

P0 MVP 目标是跑通工单系统独立闭环的最小可用版本：

```text
问题单输入 -> 人工分流 -> 创建业务需求/技术需求/缺陷工单 -> 叶子工单执行 -> 基础列表/视图 -> 基础统计
```

P0 不追求完整看板、复杂拆分、AI 能力和高级统计，而是优先验证核心对象、状态机、来源追溯和执行闭环。

## 4. P0 明确不做

以下能力不进入 P0：

| 能力 | 阶段 | 原因 |
|---|---|---|
| 问题单补充信息 | P1 | 协作增强，不影响主闭环 |
| 问题单重新打开 | P1 | 异常回退能力，首版可后置 |
| 关联已有工单 | P1 | 涉及 n:n 高级分流，首版先做一单转一工单 |
| 多问题单合并到同一工单 | P1 | 复杂来源关系，后置 |
| 一个问题单转多个工单 | P1 | 复杂分流，后置 |
| 工单二级拆分 | P1 | 涉及父子计算，后置 |
| 父工单状态/进度计算 | P1 | 依赖拆分能力 |
| 表格视图 | P1 | 高级视图 |
| 泳道看板 | P1 | 高级看板 |
| 自定义筛选/分组/排序 | P1 | 可先做基础筛选 |
| 缺陷转需求 | P1 | 重要但非首版闭环必需 |
| AI 自然语言创建工单 | P1 | 智能增强 |
| 高级统计和导出 | P2 | 管理增强 |
| AI 智能问题单分流 | P3 | 智能分流增强 |
| GienCoder 联动 | P3 | 外部联动 |

## 5. P0.1 核心业务闭环

### 5.1 问题单闭环

```text
手动创建问题单 -> 编辑问题单 -> 查看问题单 -> 人工分流 -> 转工单/关闭
```

### 5.2 工单创建闭环

```text
问题单转工单 或 手动创建工单 -> 工单详情 -> 来源追溯 -> 进入执行状态机
```

### 5.3 叶子工单执行闭环

```text
待分配 -> 待开发 -> 开发中 -> 已完成
待分配/待开发/开发中 -> 已取消
```

### 5.4 管理查看闭环

```text
工单列表 -> 基础指标 -> 分类统计 -> 状态分布统计
```

## 6. P0.1 数据模型预留

虽然 P0.1 不做拆分、AI 和高级视图，但数据模型需要预留以下字段，避免后续大改：

| 预留字段/关系 | 目的 |
|---|---|
| work_item.parent_id | P1 工单拆分 |
| work_item.level | P1 二级结构 |
| work_item.is_leaf | P1 叶子工单判断，P0 也可使用 |
| work_item.source_type | 支持问题单转入、人为创建、缺陷转需求、AI 创建 |
| work_item.source_defect_id | P1 缺陷转需求 |
| work_item.ai_creation_id | P1 AI 创建追溯 |
| work_item_source | 问题单与工单 n:n 来源关系 |
| status_log | 状态流转追溯 |
| progress_log | 进度流转追溯 |
| view_config | P0.2/P1 个人/团队视图 |

## 7. P0.1 验收口径

P0.1 可上线的最低验收条件：

1. 用户可以创建、编辑、查看问题单。
2. 待分流问题单可以转业务需求、技术需求、缺陷或关闭。
3. 转工单后，问题单状态变为已转工单，并建立来源关系。
4. 用户可以手动创建业务需求、技术需求、缺陷。
5. 工单详情可以查看来源、状态、负责人、执行人、进度等信息。
6. 未指定执行人/团队的工单为待分配。
7. 已指定执行人/团队的工单为待开发。
8. 叶子工单可以从待分配流转到待开发、开发中、已完成。
9. 叶子工单可以从待分配、待开发、开发中取消。
10. 开发中叶子工单可以更新 1%-99% 进度。
11. 完成后进度自动为 100%。
12. 取消原因必填。
13. 工单列表可以查看和基础筛选。
14. 基础指标可以展示问题单、工单、分类、状态数量。
15. 所有核心操作有操作日志或状态日志。

## 8. 架构设计输入

架构设计应优先覆盖 P0.1，并为 P1/P2/P3 预留扩展：

1. 问题单模型。
2. 工单模型。
3. 问题单与工单来源关系。
4. 工单状态机。
5. 进度记录。
6. 权限模型。
7. 基础列表与统计查询。
8. 后续父子工单、视图、AI 创建的扩展字段。
