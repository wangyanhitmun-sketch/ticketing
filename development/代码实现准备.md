## 3. 建议工程分层

```text
src/
├── domain/              # 领域类型、枚举、状态机规则
├── db/                  # 数据库连接、迁移、repository
├── services/            # issue/workItem/triage/workflow/metric 服务

├── tests/               # 单元测试、集成测试、E2E

```

## 4. 推荐实现顺序

### Step 1：基础类型与错误模型

- 复制或生成 `domain-types.ts`。
- 定义统一错误码。
- 定义统一 API 响应。
- 定义分页类型。

### Step 2：数据库迁移

- 引入 `P0-schema-postgresql.sql`。
- 根据技术栈转换为 migration。
- 确认 `teams` / `users` 是否复用已有账户系统。

### Step 3：Repository 层

优先实现：

- IssueRepository
- WorkItemRepository
- SourceRepository
- StatusLogRepository
- ProgressLogRepository
- AuditLogRepository

### Step 4：状态机服务

实现：

- 初始状态计算。
- 问题单状态流转校验。
- 工单状态流转校验。
- 进度更新校验。

### Step 5：问题单服务

实现：

- 创建问题单。
- 编辑问题单。
- 问题单列表。
- 问题单详情。
- 关闭问题单。

### Step 6：工单服务

实现：

- 创建三类工单。
- 编辑工单。
- 工单列表。
- 工单详情。
- 来源追溯。

### Step 7：分流服务

实现事务：

```text
校验问题单 -> 创建工单 -> 创建来源关系 -> 更新问题单状态 -> 写日志 -> 提交
```

覆盖：

- 转业务需求。
- 转技术需求。
- 转缺陷。

### Step 8：工作流服务

实现：

- 分配。
- 开始处理。
- 更新进度。
- 完成。
- 取消。

### Step 9：统计服务

实现：

- 基础指标。
- 工单分类统计。
- 工单状态分布。

### Step 10：API 层与测试

- 按 OpenAPI 暴露接口。
- 按 `testing/P0-测试方案.md` 编写测试。

## 5. 关键实现约束

1. 状态变更必须通过状态机服务。
2. 问题单转工单必须使用数据库事务。
3. 问题单转工单必须支持幂等键。
4. 工单分类和来源类型创建后不可普通编辑。
5. 完成工单必须将进度设为 100。
6. 开发中手动进度只能为 1-99。
7. 取消必须填写取消原因。
8. 查询和统计必须统一应用数据范围。
9. 关键操作必须写审计日志。
10. P0 中所有工单默认为 `level=1`、`is_leaf=true`。

## 6. 首批测试建议

优先自动化：

1. 问题单转业务需求并完成。
2. 问题单转技术需求并取消。
3. 问题单转缺陷并完成。
4. 手动创建三类工单。
5. 非法状态流转。
6. 来源关系回滚。
7. 权限隔离。
8. 统计口径一致性。
