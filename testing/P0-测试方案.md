## 1. 测试目标

验证工单系统 P0.1 是否完成独立闭环：

```text
问题单创建/编辑/查看 -> 人工分流 -> 创建业务需求/技术需求/缺陷工单 -> 来源追溯 -> 叶子工单执行状态机 -> 工单列表 -> 基础统计
```

测试重点：

1. 核心业务链路能跑通。
2. 状态机流转合法且可追溯。
3. 来源关系一致且可双向追溯。
4. 权限和数据范围不越权。
5. 列表、筛选、统计结果与数据一致。
6. 异常场景不会产生脏数据。

## 4. 主链路 E2E 测试

### TC-E2E-001 问题单转业务需求并完成

前置条件：

- user_submitter 可创建问题单。
- user_triager 可分流问题单。
- user_dev 可处理分配给自己的工单。

步骤：

1. user_submitter 创建问题单，线索类型为需求线索。
2. user_triager 打开待分流问题单。
3. user_triager 将问题单转为业务需求，指定 team_a 或 user_dev。
4. 系统创建业务需求工单。
5. user_dev 开始处理工单。
6. user_dev 更新进度为 50%。
7. user_dev 完成工单。
8. user_manager 查看统计。

预期结果：

- 问题单状态为已转工单。
- 工单分类为业务需求。
- 工单来源类型为问题单转入。
- 来源关系可在问题单和工单详情双向查看。
- 工单状态流转为待开发 -> 开发中 -> 已完成。
- 完成后进度为 100%。
- 状态日志、进度日志存在。
- 基础统计中业务需求数量和已完成数量正确增加。

### TC-E2E-002 问题单转技术需求并取消

步骤：

1. 创建技术类问题单。
2. 分流为技术需求，不指定执行人或团队。
3. 确认工单状态为待分配。
4. 分配给 user_dev。
5. 确认状态变为待开发。
6. 取消工单并填写取消原因。

预期结果：

- 工单状态最终为已取消。
- 取消原因、取消人、取消时间被记录。
- 状态分布中已取消数量正确。

### TC-E2E-003 问题单转缺陷并完成

步骤：

1. 创建缺陷线索问题单，填写实际结果、期望结果、复现步骤。
2. 分流为缺陷，指定执行人。
3. 开始处理。
4. 更新进度。
5. 完成。

预期结果：

- 缺陷工单保留实际结果、期望结果、复现步骤。
- 工单来源可追溯到问题单。
- 完成后状态和进度正确。

### TC-E2E-004 手动创建三类工单并流转

步骤：

1. 手动创建业务需求，不指定执行人。
2. 手动创建技术需求，指定执行团队。
3. 手动创建缺陷，指定执行人。
4. 分别验证初始状态。
5. 将三类工单推进到开发中。

预期结果：

- 无执行主体的工单为待分配。
- 有执行主体的工单为待开发。
- 三类工单都可以进入统一状态机。

## 14. 验收出口标准

P0.1 可验收上线需满足：

1. 所有 P0 主链路 E2E 用例通过。
2. 所有状态机非法流转用例通过。
3. 所有来源追溯用例通过。
4. 权限隔离用例通过。
5. 统计数据与列表数据口径一致。
6. 转工单事务失败不会产生孤立工单或孤立来源关系。
7. 关键操作日志可追溯。
8. 无 P0 阻塞级缺陷。

## 15. 风险与建议

| 风险 | 说明 | 建议 |
|---|---|---|
| P0 范围偏大 | 主链路涉及多个模块 | 优先自动化主链路 E2E |
| 权限遗漏 | 列表、详情、统计都需过滤 | 建立统一权限测试数据 |
| 状态机绕过 | 直接更新状态会破坏日志 | 所有状态动作必须走统一服务 |
| 事务一致性 | 转工单涉及多表 | 必测回滚场景 |
| 统计不一致 | 实时统计与筛选口径可能不一致 | 统计 SQL 与列表查询共用过滤条件 |
